B. FINDINGS THE ARCHITECT COULD NOT SEE (no repo access) — RECONCILE
────────────────────────────────────────────────────────────────────────
B1. *** The app ALREADY ingests TourneyMachine. ***  [GROUNDED]
    P2 has ImportSchedulePage (src/pages/ImportSchedulePage.jsx) + a
    `parse-tournament-schedule` edge function + useImportSchedule commit flow
    (parse → validate → dedup → bulk upsert of events). The spec frames F2 Phase 1
    as "P1's scraper pushes to a NEW ingest route." That risks a SECOND TourneyMachine
    ingestion path inside the very system the IA is trying to de-duplicate.
    CC RECOMMENDATION: converge. Either (a) `ingest.syncProgramData` reuses the
    existing `parse-tournament-schedule` parser + commit pipeline (preferred — one
    TM parser, one upsert path), or (b) the spec explicitly states why a second,
    automated path coexists with the human paste path and how they don't collide on
    the same events. Architect to rule.

B2. DATA-MODEL MAPPING GAP (scraper shape ≠ app shape).  [GROUNDED]
    P1's scraper emits FLAT denormalized games (homeTeam, awayTeam, homeScore,
    awayScore, gameTime, division, externalId, isLegacyHome, legacyWon). P2 stores
    normalized: game_results.event_id → events (the org's own event) + opponents
    (FK). So ingestion must, per scraped game: resolve/create the matching EVENT,
    match/create the OPPONENT, then upsert game_results. The spec's idempotency key
    "(tournamentId, divisionId, gameId)" lives on the SCRAPER side; on the app side
    the natural key is event_id (+ an external_game_id column to dedupe re-runs).
    CC RECOMMENDATION: the Phase 1 DDL needs an external-id column (e.g.
    game_results.external_game_id UNIQUE-per-org) for idempotency, and the spec
    should specify the event/opponent resolution rule (auto-create vs match-only).
    gameId ≟ scraper externalId — CC to confirm in code.

B3. game_results lacks a DIRECT org_id (RLS walks event→team→org).  [GROUNDED]
    Fine for reads today, but a service-auth ingest path is cleaner with a direct
    org_id + a BEFORE INSERT trigger deriving it from the event, and a simplified
    RLS. CC RECOMMENDATION: add game_results.org_id (NOT NULL, indexed) in the
    Phase 1 migration; low-risk, tightens the ingest scoping.
