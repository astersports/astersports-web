5. WHY THE ORIGINAL AUDIT MISSED THIS (the meta-gap, owned)
════════════════════════════════════════════════════════════════════════
The 2026-06-25 audit was scoped to CODE + the Supabase DB. It never asked, per repo,
"does `main` deploy, and is the platform it deploys to alive?" The signals were all
present and not connected: every Forge repo's `template.json` + `server/_core` Forge SDK;
PR #103 literally titled "Migrate off Manus → Railway"; Frank's own "migrate St Patrick
off Manus" note. The audit even named the exact pattern — X1 "repo↔prod drift" — but
only checked the DATABASE side of "is prod running the code," never the HOSTING side.
The discipline fix: an audit of "are the repos set up properly" MUST include a
deployment-wiring pass — for each repo: deploy target, is it alive, does main serve,
where's the DB, what platform dependencies are in the critical path. Added to the
verify-before-claiming-clean checklist.
