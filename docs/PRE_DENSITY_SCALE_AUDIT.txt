PART 1 — THE LOGIN FAILURE ("OAuth callback failed")  ← do this first
------------------------------------------------------------------------------------
What happens: Google login completes, then the app shows {"error":"OAuth callback
failed"} when it tries to write your user row.

What I proved (so we don't chase ghosts):
  - I reproduced the EXACT failing database insert (same drizzle + postgres-js +
    prepare:false, same SQL, same parameters) against a real Postgres locally.
    It PASSED. The timestamp is serialised correctly — the earlier "Date" theory
    was WRONG, and I did NOT ship a fix for a non-bug.
  - In production every app DB connection goes through `user=pgbouncer` (the
    Supabase TRANSACTION pooler, port 6543), and Postgres logged NO error for the
    failed insert. So the failure is at the pooler/driver layer, not the query.

THE FIX (Frank, ~2 min):  switch DATABASE_URL to the SESSION pooler.
  - The transaction pooler (port 6543) is meant for serverless. A long-running
    server like Railway should use the SESSION pooler (port 5432).
  - Supabase dashboard -> Connect -> Connection string -> choose "Session pooler"
    (port 5432, host ends in ...pooler.supabase.com) -> copy it, put your DB
    password in -> replace DATABASE_URL in Railway -> redeploy.
  - No code change needed (prepare:false is harmless on the session pooler).

Backstop I shipped (commit c3aa5c9): the OAuth callback now logs the REAL underlying
cause instead of swallowing it. So if the session pooler does NOT fully fix it, the
next login attempt will print "[OAuth] Google callback failed -> ... cause ..." in
the Railway deploy logs — paste me that line and I'll know exactly what to fix.

    the in-process cron poller (ENABLE_SCHEDULER is already on, so jobs resolve).
    If you leave it OFF, generation runs on the synchronous SSE path (up to 180s,
    holds the request open) — works on Railway, just less robust.
  - The live flags — YOU flip these by hand, one at a time (CLAUDE.md §1/§3):
      STUDIO_DENSITY_LIVE=true         (density v1 "thin")

      STUDIO_SCALE_LIVE=true           (scale)
    These REQUIRE STUDIO_MASK_PROVIDER=sam2 (already set) or the app refuses to
    boot — that's the intended guard against charging for an op that can't run.
  - After flipping, confirm posture: GET /api/studio/posture (cron-secret auth)
    should show the flag live and replicateConfigured:true.
