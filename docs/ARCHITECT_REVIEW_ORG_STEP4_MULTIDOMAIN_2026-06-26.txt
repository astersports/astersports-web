1. WHAT THIS BUILDS (spec §4; architect sign-off refinements applied)
--------------------------------------------------------------------------------
Multi-domain lock: an org's allowed-domain lock is now a SET, auto-seeded from the
owner's email at provisioning, super-admin-extendable, public-domain-guarded, and
enforced server-side on the invite path. Plus the D2 ruling (role ceiling).

- shared/domain.ts (PURE, fully unit-tested — server/domain.test.ts, 14 cases):
    · PUBLIC_EMAIL_DOMAINS denylist + isPublicEmailDomain — the load-bearing guard.
    · autoDomainLockForOwnerEmail(email) → the owner's domain, or NULL if public
      (→ no auto lock; invite-only). NEVER auto-locks gmail/outlook/icloud/…
    · emailAllowedForDomains(email, set) — set-aware membership; empty set = allow
      all (explicit per-email invites still govern), parity with the legacy null.
    · normalizeDomain (accepts "@d"/"d"/"a@d"); emailAllowedForDomain kept for
      back-compat.
- drizzle: new tenant_domains table (id, tenantId, domain, verifiedAt NULL,
  createdAt) + UNIQUE(tenantId,domain) + tenant index. Migration 0004.
- server/tenantDomains.ts: listTenantDomains / addTenantDomains (idempotent via
  onConflictDoNothing) / isEmailAllowedForTenant (set when present, else legacy
  single-column fallback — existing orgs keep working until migrated).
- WIRING:
    · platform.provisionFirm auto-seeds the set from the explicit domainLock + the
      owner's email domain (public-guarded). (Audit metadata already records it.)
    · tenants.invite now enforces the SET (isEmailAllowedForTenant) AND the D2 role
      ceiling: an admin cannot invite an admin — FORBIDDEN; only the owner grants
      the admin role. Members can still be invited by admins.

2. STORAGE FORK — normalized table (CC lean, built; architect please ratify)
--------------------------------------------------------------------------------
Two shapes for "a set of domains per org":
  A. NORMALIZED tenant_domains table (one row per (tenantId, domain)) — BUILT.
     + clean join/exists enforcement; + natural home for the future per-domain
     verifiedAt (verified-domain flow the sign-off mentions); + UNIQUE index gives
     idempotent adds; − one more table.
  B. jsonb/text[] array column on tenants.
     + no new table; − no per-domain verified_at without nesting; − array-contains
     enforcement is clumsier; − re-introduces the "config on the row" pattern.
CC LEAN: A. The verifiedAt column + clean enforcement + idempotent adds make the
table the more durable choice, and it isolates cleanly under RLS. Legacy
tenants.allowedEmailDomain is RETAINED as the fallback (no data migration in this
PR; existing single-domain orgs keep working). If you prefer B, say so — it's a
contained swap of tenantDomains.ts + the migration.

3. SEMANTICS (so the enforcement contract is explicit)
--------------------------------------------------------------------------------
- Tenant with a NON-EMPTY domain set: invite/join allowed iff the email's domain
  ∈ set. (Explicit per-email invite is still the admin action; there's no public
  self-join surface live yet.)
- Tenant with an EMPTY set AND no legacy domain: any email may be invited (admin-
  gated). This is the "invite-only, no domain auto-join" state a public-domain
  owner lands in (autoDomainLockForOwnerEmail returned null).
- Public-domain owner at provisioning → NO auto lock (guarded). A super-admin may
  still explicitly pass a domainLock; explicit super-admin input is NOT public-
  guarded (only AUTO-set is), matching "super admin edits any org's domains".
