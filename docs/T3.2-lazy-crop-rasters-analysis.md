# T3.2 — Lazy Crop-Sized Instance Rasters: Analysis

**Status:** Analyzed, deferred to a dedicated refactor session.  
**Date:** 2026-06-23  
**Reason for deferral:** The acceptance criterion requires "output byte-identical" which needs careful integration testing with real images. The refactor touches the core `InstanceMask` type contract and all consumers.

---

## Current State

Instance rasters are produced at **crop size** in `sam2Mask.ts` (`instancesFromMasks`), then immediately expanded to **full-image size** via `remapRasterToFullImage` in `sam2Provider.ts` (line 254):

```ts
raster: inst.raster
  ? { width: s.width, height: s.height, data: remapRasterToFullImage(inst.raster.data, s.cropWidth, s.cropHeight, s.width, s.height, s.bbox) }
  : undefined,
```

For a 40MP image with 200 instances, this allocates 200 × (40M bytes) = **8 GB** of raster data.

## Consumers That Assume Full-Image Rasters

1. **`densityThin.ts`** (line 38): `if (r && r.width === w && r.height === h)` — indexes `r.data[i]` with full-image `i`.
2. **`densityRedistribute.ts`** (line 197): Same dimension check + full-image indexing in `markInstance`, `centroid`, `instanceBBox`, `cropAlpha`.
3. **Eval harnesses**: `buildLabelMap.ts` indexes instance rasters at full-image coordinates.

## Proposed Approach (for future session)

1. Add `origin: { left: number; top: number }` to `RasterMask` (default 0,0 = full-image).
2. Keep instance rasters at crop size in `sam2Provider.ts` — set `origin` to the crop offset.
3. Add a `lazyRemap(inst, fullWidth, fullHeight): Uint8Array` helper that remaps on demand.
4. Update `markInstance` in both ops to use origin-aware indexing:
   ```ts
   const ox = inst.raster.origin?.left ?? 0;
   const oy = inst.raster.origin?.top ?? 0;
   for (let cy = 0; cy < inst.raster.height; cy++) {
     for (let cx = 0; cx < inst.raster.width; cx++) {
       if (inst.raster.data[cy * inst.raster.width + cx] > 127) {
         mask[(oy + cy) * w + (ox + cx)] = 1;
       }
     }
   }
   ```
5. Validate output byte-identical via the eval bench (`pnpm eval`).

## Expected Memory Savings

- Before: 200 instances × 40M bytes = ~8 GB peak
- After: 200 instances × (cropW × cropH) bytes ≈ 200 × 500K = ~100 MB peak
- **~80× reduction** for the pathological case.

## Risk

- Any consumer that indexes `inst.raster.data` with full-image coordinates will silently produce wrong results if not updated.
- The `densityRedistribute` dim-drift guard (line 197) will need updating to check crop-sized dimensions instead.
