2. THE STRATEGY (decided — do not re-derive; full text in DENSITY_SCALE_STRATEGY.txt)
================================================================================
2.1 The client's catalog splits into FOUR surface buckets (product names are diagnostic):
  A. SCATTERED all-over prints ("tossed/trailing floral", dots) — artfully RANDOM, NOT a
     geometric repeat. High volume. → DENSITY'S SWEET SPOT.
  B. PLACED embellishment / sequins / paillettes (brand signature) — hard case.
  C. SOLIDS — density/scale DO NOT APPLY (UI must suppress the controls).
  D. LACE.

2.2 The key insight that reorganizes everything:
  Their prints are SCATTERED, not periodic. That splits the two ops in OPPOSITE directions:
  - DENSITY loves scattered: counting motifs + removing an exact fraction is
    periodicity-agnostic. Their florals are ideal for density.
  - SCALE chokes on scattered: the old repeat guard (FFT + autocorrelation) looks for a
    PERIODIC tile; a tossed floral has no strict period → it gets FALSE-REJECTED (~1/3 of
    genuine repeats — the CLAUDE.md §4 G3 complaint). FIX (Phase 2): scale must treat
    "all-over scattered coverage" as scalable and scale by RESIZING MOTIFS, not re-tiling.

2.3 The architecture (combine, don't pick):
  - FORMULA/DETERMINISTIC = ACCURACY. Only it can count + place EXACTLY ("remove exactly
    50%"). Billable, reproducible. The spine.
  - GENERATIVE AI = REALISM. Only it makes a hard garment look photographically right.
  - THE TRICK: run the generative pass MASKED so it polishes the LOOK without changing the
    COUNT/GEOMETRY. Route the finish to the garment: solid ground → cheap formula fill;
    busy/complex ground or on-model → constrained generative finish.

2.4 FLUX.1 Fill spike (PR #104) — EVALUATED, NOT ADOPTED (closed/parked). FLUX ≈ the
  simple fill on solid grounds (no visible win), ~100x cost, new sub-processor + paid
  licence. Reopen only if a client garment has a genuinely busy/photographic ground.

4. THE THREE PHASE-1 METRICS + RESULTS (rigorous bar = synthetic, exact known N)
================================================================================
M1 — Density count fidelity (pipeline removes the requested fraction within ±10%):
  100% on synthetic (8/8 at 30% & 50%, N=30/60/90/120). The deterministic remover takes
  exactly round(N·p/100) given correct instances — robust to overlap/colour/contrast.
  CAVEAT: M1 is CONDITIONAL on correct instances; alone it proves the math, not detection.

M2 — Segmenter detection (count within ±10% of known N):
  - Clean discrete synthetic (well-separated): 100% (3/3, exact).
  - HARD conditions (the adversarial audit, eval/phase1Audit.ts): classical = 4/10 = 40%.
    It collapses on heavy overlap (23% err), multicolor-bunched (23%), two-tone ground
    (63%), low-contrast (100% — finds nothing). THIS IS THE CLASSICAL CEILING → needs SAM2.

M3 — Scale detector recall (accept all-over/scattered; 0 false-accepts on placed/border):
  22% → ~95% after the coverage reframe, with 0 false-accepts maintained. Only miss is a
  synthetic checkerboard stress case (not a real garment print). Locked in CI via the
  detectorAccuracy corpus (scattered + gradient fixtures; accept floor raised 4→8) and
  repeatDetector unit tests (coverage path, gradient-reject, placement-reject).

Honest reckoning (after Frank challenged the headline): the early "90%" for M1/M2 was on
EASY synthetic and over-stated readiness. The TRUE picture: density MATH (M1) and scale
DETECTION (M3) are robust offline wins; offline COUNT accuracy on real bunched/multicolor/
low-contrast florals is ~40% — NOT billing-grade. That gap is what the SAM2 eval (§5) closes.

5. THE SAM2 GO/NO-GO MEASUREMENT (docs/SAM2_COUNT_EVAL.md) — THE DECISIVE RESULT
================================================================================
Question: does the REAL production SAM2 count path hit ±10% on the hard conditions where
classical hit 40%?  Measured via a CI workflow (creds via the REPLICATE_API_TOKEN secret;
sentinel-triggered): autoSegment(dataUrl, params) → finishSam2Segmentation(seg, meta) →
DERIVED instances (production derivation: specks <0.02% dropped, giants >20% crop dropped
as ground, cap STUDIO_MAX_INSTANCES=200) → count. NOT raw mask count. Best of pps16/32/64,
no per-image cherry-picking, no fitting to N. The vision-LLM fabric LOCATE is bypassed (full
frame for synthetic, manual bbox for reals) so it measures SAM2's COUNTING, not the locator.

RESULT — SYNTHETIC (rigorous, exact N): SAM2 = 10/10 within ±10%. Per condition:
  heavy-overlap   classical 23% → SAM2 8%
  multicolor-bunch classical 23% → SAM2 3%
  two-tone ground  classical 63% → SAM2 2%
  gradient ground  classical 12% → SAM2 3%
  low-contrast     classical 100% (blind) → SAM2 2%;  contrast-mid 100% → 0%
  (separated/high-contrast controls all 2-3%)

VERDICT: BILLING-GRADE DENSITY IS ACHIEVABLE — with SAM2 as the count source. SAM2 does
what classical fundamentally cannot: split touching motifs, and see low-contrast / varied-
ground prints. This is the go.

CAVEAT (do not skip): the REAL-floral rows are NOISY and NOT trustworthy yet —
  poppette SAM2 45 vs classical 93 (UNDER), pindot 1 (found ~nothing), tossed 143,
  tessa 211, stassie 130 — because (a) no firm hand ground-truth, and (b) the eval fed a
  MANUAL bbox crop, not the production vision-LLM locate. So the real-image path needs the
  full production locate + crop + per-route tuning before billing. The trustworthy signal
  is the synthetic known-N bar (10/10).

To re-run: bump `eval/RUN_SAM2_EVAL` (the "run:" line) and push on a claude/** branch →
the workflow runs (~42 SAM2 calls, ~pennies each) and commits docs/SAM2_COUNT_EVAL.md +
eval/out/sam2-count/ overlays back to the branch. Cost ~$0.1–0.4/run.
