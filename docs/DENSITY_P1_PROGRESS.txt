2. P1 PREVIEW OP (the "+ 10/20/50% preview" deliverable).
   server/_core/studio/ops/densityPreview.ts + server/densityPreview.test.ts (9 tests).
   - Pins the "% drop" semantics: density is COUNTING. "remove p%" = remove
     round(N * p/100) motif INSTANCES by count; survivors stay byte-identical.
     Periodicity-agnostic -> exact + previewable on scattered prints (strategy §1).
   - TRUTHFUL: every preview step is produced by the SAME deterministic densityThin
     the billed op runs, so the p% a user previews is byte-identical to what they get
     when they pick p%. (Test asserts this.)
   - NO-OP HONESTY (§1 "never bill for a no-op"): each step reports requestedRemoval
     AND actual removed; on a densityThin degrade (missing/empty raster, no bare
     ground, instance dim-drift) removed=0 + noop=true. Caller must not bill.
   - Pure/deterministic; carries raw RGBA, no encode in the op.

3. OFFLINE MOTIF SEGMENTER (eval-only, to demo P1 on a REAL garment without SAM2).
   eval/segmentMotifs.ts + eval/densityPreviewDemo.ts.
   - Classical contrast+connected-components stand-in: estimate ground (k-means/Lab)
     inside a fabric bbox, threshold far-from-ground pixels as motif, CC -> instances,
     area-filter, emit an instance label-map PNG that evalMaskIO already understands.
   - Lets densityPreview run end-to-end offline on a real piece. Demonstrated:
       LIKELY Poppette (black dots/grey): 93 motifs detected -> -10% remove 9,
         -20% remove 19, -50% remove 47. Visual: ~half the dots erased + ground-filled,
         survivors intact (eval/out/density-preview-poppette.png).
       LIKELY Tessa (roses/dark): 112 -> remove 11/22/56.
   - Flat LAB fill leaves faint patches on the pinstriped Poppette ground — exactly
     the textured-ground case the strategy doc (§3) says LaMa texture-fill would
     polish (and notes LaMa is currently 404'ing in prod — latent bug, low priority).

DESIGN LEAN TO RATIFY (Architect-scoped — flagged, not unilaterally shipped to prod)
-----------------------------------------------------------------------------------
The eval segmenter is deliberately EVAL-ONLY (under eval/, not wired to the mask
provider or router) because mask-provider is Architect-scoped (CLAUDE.md §2.4) and a
sub-processor-adjacent surface. Open question for the Architect:
  Should a hardened classical motif counter become a real production InstanceMask
  source for HIGH-CONTRAST scattered prints — a no-subprocessor fast path that avoids
  a SAM2 call when contrast is clean — with SAM2 as the fallback for hard/low-contrast
  grounds and on-model isolation?
  CC's lean: YES as a guarded fast path (cheaper, no sub-processor for the easy cases),
  but it does NOT replace SAM2 for the count GATE — G3 count accuracy must still be
  proven on SAM2 instance masks. Decision deferred to the Architect.
