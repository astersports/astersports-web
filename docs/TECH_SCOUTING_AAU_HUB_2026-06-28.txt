WHAT THIS PILLAR IS FOR (one line): the public, mobile-first parent/coach surface for
finding AAU tournaments, tracking a kid's team across events, and seeing schedules /
standings / live scores / advancement — a thin public read-consumer of the aster-sports
Supabase backbone.

HARD INVARIANTS THIS PILLAR MUST NEVER BREAK (the ones that apply here):
  · Honest claims — labels/copy/UI must never contradict the data (the exact-vs-estimate
    constitution: no model likelihood shown before it earns calibration; a clinched team
    never shows a probability; null → graceful absence, never a fabricated value).
  · Guardian-verified COPPA gate on Film — canAccessChild stays false for everyone; no
    minor-face recognition; child video AI is always its own gated, privacy-designed path.
  · Stripe webhook is the sole writer of is_entitled (Aster Plus). The hub never writes
    entitlement; the paywall is read-side only.
  · Schema migrations are owner-applied only (the backbone is aster-sports).
  · Public reads are RLS- + org_is_public_listed-gated; the anon key is publishable.

CORE FLOWS [GROUNDED]
  · Discovery/search: `search_public_aau` / `search_public_teams` RPCs use
    `lower(display_name) LIKE '%q%'` substring matching — no ranking, no typo tolerance,
    no FTS/trigram/vector [GROUNDED aster-sports …regrain_c.sql:122,196,207,249].
  · Team identity: the qualified key `qkey = resolved_key:gender:grade` is the INTERIM
    de-dup (ruling C; applied this session). It strictly refines a name-slug but is NOT a
    canonical entity — "High Rise - Brie" vs "High Rise (CT) - Brie" stay separate by luck
    of the slug, and one real team across tournaments is grouped only because program·
    grade·gender happens to be stable. Canonical team entity = the explicitly-planned next
    substrate (B) [GROUNDED migration header …regrain_c.sql:23-25].
  · Advancement predictor: `client/src/lib/standings/predictBracket.ts` — deterministic,
    enumerates every remaining win/loss combo, computeStandings per combo. Each game is
    strength-weighted by `eff` (a logistic of opponent-adjusted margin ratings,
    RATING_SCALE=0.1, predictBracket.ts:71-75). The strength-weighted oddsPct is
    DELIBERATELY SUPPRESSED in the UI until calibration on real results ("A4"); the card
    leads with the exact count (BracketOdds.tsx:50-60). Flagged limitation: simulated
    games use a nominal 1-0 margin, so margin-dependent tiebreaks resolve only at
    win/loss granularity (predictBracket.ts:16-17).
  · Strength rating `eff`: opponent-adjusted average capped margin (the standings
    substrate). A hand-rolled SRS-lite, not a principled rating system [GROUNDED
    aster-sports standings RPCs + predictBracket].
  · Live scores: `NextGame` polls on a 30s interval (setInterval 30_000); the data source
    is a TourneyMachine ingest that is effectively FINAL-ONLY, so a game can read "live"
    with no score for its whole duration [GROUNDED NextGame.tsx; this-session review].
  · Ingest: a TourneyMachine HTML scrape (edge function). The "generic adapter / any link"
    capability is NOT built — non-TourneyMachine links hard-reject (copy fixed this session
    to stop overpromising). Live coverage gap, not just architectural [GROUNDED PastePanel +
    validator].
  · Film: verified-guardian COPPA gate, canAccessChild=false for all, AI-review gated
    [GROUNDED FilmAiReviewGate / entitlement.ts].
  · Money: paywall CTA opens the Plus gate; checkout not wired; is_entitled is
    Stripe-webhook-only [GROUNDED].

WEAKEST / MOST HAND-ROLLED (the scouting targets)
  1. Search = LIKE substring (no typo tolerance, no ranking). [GROUNDED]
  2. Team identity = interim qkey, no canonical entity. [GROUNDED]
  3. Strength rating = hand-rolled avg-margin; predictor margin limitation. [GROUNDED]

C1 — Route/app code-splitting + manualChunks (separate the hub from the Studio bundle)
  · What: Vite manualChunks + React.lazy route splitting so the AAU hub loads only its

  · Fits: the single >500 KB entry chunk [GROUNDED build output]; `AAUBasketball.tsx` and

  · Win: FCP/TTI on 4G (§16.10 budgets the hub already declares). Concrete, measurable.

  · Privacy: none (no data leaves).
  · GATE: 🟢 ROUTINE. Collision: none.
  · Lean: PROTOTYPE → ship. Highest ROI / lowest risk in the set.

C2 — Typo-tolerant + ranked search via Postgres pg_trgm (replace LIKE substring)
  · What: pg_trgm trigram similarity + GIN index; rank by similarity(); keep LIKE as a
    fast-path. Supabase ships pg_trgm. [GROUNDED — pg_trgm is core Postgres contrib;
    Supabase-native. VERIFY the extension is enabled on the project before pilot.]
  · Fits: `search_public_aau` / `search_public_teams` LIKE arms [GROUNDED …regrain_c.sql].
  · Win: "highrise"/"hi rise"/"High Rize" all find "High Rise"; results ranked, not
    arbitrary. Directly attacks the architect's "search disambiguation is thin" finding.
  · Cost: one migration (extension + index) on the backbone (owner-applied); no new dep,
    no sub-processor, on-DB latency only.
  · Privacy: none (no data leaves; public team names).
  · GATE: 🔴 HUMAN (changes user-visible search results + a schema migration on the
    backbone → owner-applied). Low-risk 🔴. Collision: none — deterministic, on-DB.
  · Lean: PROTOTYPE the migration + bench recall on real queries vs the LIKE baseline.
  · NOTE: pgvector/embedding *semantic* search is PARK — overkill for 2–4-word team names,
    and it adds an embedding sub-processor (privacy surface) for little gain over trigram.

C3 — Canonical team-identity entity resolution (qkey interim → stable team entity)
  · What: a real entity-resolution pass — deterministic blocking (gender·grade·circuit) +
    trigram/Jaccard similarity on normalized names, with conservative clustering that
    ERRS TOWARD SPLIT (never merges two distinct programs). Optional: small text-embedding
    similarity ONLY for the residual hard cases, computed batch/offline, not per-request.
  · Fits: the interim qkey [GROUNDED …regrain_c.sql:23-25]; unblocks `get_public_aau_team_
    schedule`, cross-tournament rating, the planned by-kid layer, film attribution, and
    5th→6th cross-season aging.
  · Win: one stable identity per real team across tournaments/seasons → fixes the
    "tracked teams indistinguishable" + cross-season problems at the root.
  · Cost: scraper/backbone-side resolution + a canonical_team table + backfill. Moderate.
    Deterministic+trgm path = no external service. Embedding path = batch embeddings
    (sub-processor) only if trgm is insufficient.
  · Privacy: deterministic/trgm path = none. Embedding path = team NAMES (not PII) to an
    embedding provider; still warrants a sub-processor note.
  · GATE: 🔴 HUMAN (changes user-visible grouping + schema; architect-scoped fork —
    already the "B" substrate). Collision with honest-claims IF it ever merges two real
    teams → hard rule: precision-over-recall, err toward split (matches ruling C).
  · Lean: PROTOTYPE deterministic+trgm; PILOT embeddings only if the residual justifies a
    sub-processor. This is the architect's already-named next substrate — sequence it.

C4 — Principled strength rating (Glicko-2 / TrueSkill) to replace hand-rolled `eff`
  · What: swap the avg-margin `eff` for a rating system with uncertainty (Glicko-2 or
    TrueSkill-style), updated from real results, feeding both standings rating and the
    predictor's win-prob. [GROUNDED — Glicko-2 (Glickman) and TrueSkill (Herbrich et al.)
    are published, well-established; no benchmark asserted here — to be measured.]
  · Fits: `eff` in predictBracket.ts:71-75 + the standings rating substrate.
  · Win: better-calibrated win probabilities (handles small samples + opponent strength +
    uncertainty), which is the prerequisite for ever un-suppressing the odds %.
  · Cost: pure-function rating module + an eval; no external service, negligible latency.
  · Privacy: none.
  · GATE: 🟡 BENCH — model swap; must beat the current eff in the offline harness BEFORE
    proposal. Un-suppressing any UI % remains a separate 🔴.
  · ACCEPTANCE BAR (declare before run): on a held-out set of completed AAU divisions,
    the new rating's game-level win prediction must beat current-eff on log-loss AND not
    regress accuracy; re-run the current eff baseline in the SAME pass (no stale numbers).
  · Lean: PILOT in `eval/`.

C5 — Calibrate the advancement probability (earn the suppressed % — the "A4" work)
  · What: Platt scaling / isotonic regression on the strength-weighted oddsPct against
    realized advancement, with a reliability diagram. ONLY un-suppress the % once it's
    demonstrably calibrated. [GROUNDED — Platt/isotonic are standard calibration methods.]
  · Fits: the suppressed oddsPct (predictBracket.ts:148; BracketOdds suppression).
  · Win: the hub can finally show an honest "% to advance" — the headline feature that's
    deliberately dark today.
  · Cost: calibration fit + storage of the calibrator; small.
  · Privacy: none.
  · GATE: 🟡 BENCH to fit/measure; 🔴 HUMAN to un-suppress (changes user-visible output and
    touches the exact-vs-estimate constitution head-on — Frank flips, after the bar).
  · ACCEPTANCE BAR (declare before run): expected calibration error (ECE) below a
    pre-set threshold AND Brier score beats the exact-count-only baseline, on held-out
    tournaments; if it can't calibrate, it STAYS suppressed (the failure mode is "remain
    honest," which is acceptable, not a regression).
  · Collision: this is the SANCTIONED path to satisfy honest-claims, not break it. Depends
    on C4. Lean: PILOT after C4.

C6 — Margin-aware bracket tiebreak simulation (fix the nominal-1-0 limitation)
  · What: when a remaining game feeds a margin-dependent tiebreak, sample/model a plausible
    margin (from the rating gap) instead of a flat 1-0, so the enumerated scenarios respect

  · Fits: predictBracket enumeration.
  · Win: correct if/then scenarios in divisions decided by point-diff tiebreak.
  · Cost: moderate (enumeration becomes a small Monte-Carlo on tiebreak-relevant games).
  · GATE: 🟡 BENCH (changes the displayed exact counts → also 🔴 to ship). Collision: must

    must change to reflect sampling, or keep exact counts and only refine ordering. Declare
    which before building. Lean: PROTOTYPE + bench; lower priority than C4/C5.
