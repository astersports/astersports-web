0. TL;DR — WHAT astersports.io IS NOW
════════════════════════════════════════════════════════════════════════
astersports.io is the PRODUCT-PLATFORM HUB for Aster Sports (Olive Juice Inc). It is
NOT a single-service agency page anymore — as of this session it's a multi-product
platform whose landing page is driven by a typed SERVICE REGISTRY, so each product
"lights up" as it ships. The visual language is "Celestial Cartography": a navy night
sky where live services are lit stars in a constellation and not-yet-shipped ones are
faint stars in the same sky.

URL MAP (one app, several surfaces):
  astersports.io            → marketing hub (this rebuild)
  astersports.io/studio     → Print Studio (the money-path product; separate CC lane)
  astersports.io/platform   → Studio platform console
  astersports.io/aau        → AAU basketball portal (public, gold-themed)
  astersports.app           → the youth-sports OPERATIONS app (separate repo; system of record)

DEPLOY: astersports-web auto-publishes from `main` via Manus/Forge (+ Railway). No
Vercel project for THIS repo (Vercel is the aster-sports app's host, not this one).

3. STRUCTURE — the registry is the single source; everything derives from it
════════════════════════════════════════════════════════════════════════
client/src/lib/services.ts  — THE SINGLE SOURCE OF TRUTH for the platform surface.
  ServiceEntry { id, label, navLabel, status, accent, icon, href, blurb, cta, inNav... }
  STATUS_META: status → pill label/treatment ('live' | 'beta' | 'members' | 'soon').
  PRODUCTS:  studio (live, nav "Studio") · app (beta, nav "App") · aau (live, nav
             "Programs") · websites (soon).
  SERVICES:  print-design · web · brand · strategy   (the agency "we also build" band).
  NAV_PRODUCTS = PRODUCTS.filter(inNav); ALL_SERVICES = [...PRODUCTS, ...SERVICES].
  → To bring a new service online or flip a status, edit ONE registry entry. Header nav,
    mobile nav, footer, and the constellation tiles all derive from it. No page surgery.

client/src/pages/Home.tsx  — the hub, composed of registry-driven sections:
  Header (66) → HeroSection (207, starfield + glowing mark + gradient headline) →
  PlatformSection (342, id="platform": ConstellationNode(315)+NodePill(296) render
  PRODUCTS as lit/faint star tiles) → ServicesSection (386, id="services": the agency
  band from SERVICES) → ProcessSection (470) → AboutSection (533) → FAQSection (628) →
  ContactSection (697) → Footer (849). StarAccent(17) is the shared sky motif.

client/src/pages/AAUBasketball.tsx  — the /aau portal (PUBLIC). Composes the AAU
  components below; gold theme via root var override; mounts <WeatherCard/>; "Open in
  the app" CTA → astersports.app; mobile bottom-nav (Scores/History/Records/Film).

client/src/components/aau/  — StatHeroBar, LiveScores, TournamentHistory,
  SeasonLeaderboard, FilmHighlights, Locations, Mission, SectionHeading, and
  weather/{WeatherCard, WeatherIcon, GameWeather}.

server/routers.ts  — tRPC. AAU READS are public; WRITES stay owner-gated:
  games.list / games.live / games.completed  = publicProcedure   (89/99/109)
  leaderboard.get                            = publicProcedure   (149)

  weather.home / weather.get                 = ownerProcedure     (131/136)
server/weather.ts  — Open-Meteo client (keyless, SSRF-safe, 10-min cache, stale-on-fail);
  isValidCoord now re-exported from @aster/weather (the #108 convergence).

4. SYNCHRONIZATION MODEL — "one source, many consumers"
════════════════════════════════════════════════════════════════════════
The platform thesis: each real-world concern has ONE authoritative source; every
surface CONSUMES it. Two live instances of this:

A) WEATHER → @aster/weather (the shared package; P5 repo astersports/aster-weather).
   v0.1.0 published and CONSUMED by astersports-web (#108: WMO text + coord validation).
   Weather still exists in THREE codebases historically (the drift the package removes):
     W-a  st-patricks-armonk  — the ORIGINAL, pure/decoupled, ~1050 LOC + animated icons.
     W-b  aster-sports (the app) — native build merged ~2026-06-18 (#1075–77).
     W-c  astersports-web /aau — now partially converged onto the package (#108).
   STATE: only astersports-web consumes the package so far. W-a (St Patrick) and W-b
   (the app) still carry their own copies → the convergence backlog (see §5/roadmap).
   What stays local on purpose (richer than v0.1.0): animated SVG icons, the combined
   single-call forecast, UV/wind-direction/venue-timezone. A later @aster/weather grows
   those into the shared surface.

B) GAMES / RECORDS → the aster-sports app is the SYSTEM OF RECORD; astersports.io READS.
   astersports.io must never be a second source of game truth. The frozen public read
   contract is the RPC get_public_team_records (see §6). Today /aau still renders
   scraper-backed data IN this repo; wiring it to the app's RPC is the deferred consumer
   pass (§6 step 2). That cutover is what makes "one source" literally true for scores.

PRIVACY-BY-ARCHITECTURE: astersports.io publishes NO roster. The public read projects
display columns only (no player PII). AAU is exposed fully public AS-IS per the owner's
on-record decision (PR #107) — that publishes a girls' youth team's live venue/times +
first names; the safer redacted option was offered and not chosen. Write/refresh stays
owner-gated.

6. GAME + SCHEDULING SYNC-UP (the detailed picture — read before touching ingest)
════════════════════════════════════════════════════════════════════════
ARCHITECTURE — DR-A=A1, "one writer per concern" (BUILD SPEC v3). TourneyMachine (TM) is
the external source. It is parsed ONCE, by the SCHEDULE path. Scores are a SEPARATE
writer that consumes already-identified games. The two never overlap:

   TourneyMachine
        │  (parsed once)
        ▼
   [SCHEDULE PATH]  parse-tournament-schedule  →  events            (tournaments/schedule)
        │
        └── games already identified as events
                 │
                 ▼
   [SCORES PATH]   ingest-game-results  →  game_results             (scores only; NO TM parse)
                 │   match-only to events (unmatched → reviewFlags, never phantom inserts)
                 ▼
   get_public_team_records(p_team_id)  →  astersports.io/aau        (public read; DR-B)

LIVE STATE OF F2 PHASE 1 (aster-sports project vrwwpsbfbnveawqwbdmj) — ALL DONE this
session, do NOT redo:
  ✅ Migration applied to prod (PR #1094, merged): game_results +external_game_id
     +org_id (NOT-NULL via BEFORE-INSERT derive-trigger event→team→org; existing rows
     backfilled) +UNIQUE(org_id,external_game_id) for scraper idempotency;
     feed_source_mapping table +org-scoped RLS (explicit with_check); get_public_team_records
     SECDEF RPC. Advisor-clean (0029 fix: trigger fn EXECUTE revoked from authenticated).
  ✅ Edge fn ingest-game-results DEPLOYED — v1, ACTIVE, verify_jwt=false. Auth = shared
     secret app_secrets.ingest_secret (Bearer), service-role client, no Deno.env secret.
  ✅ app_secrets.ingest_secret PROVISIONED (64-char value; lives ONLY in app_secrets).
  ⚠️ Do NOT re-apply the migration, re-provision/ROTATE the secret (rotating breaks the
     future caller unless coordinated), or redeploy unless the fn code changes.
  ⚠️ Runtime gateway smoke (live 401/200) NOT run — CC env egress policy blocks
     *.supabase.co. Verify with a curl from a normal network if desired.

THE REMAINING WORK (§5a consumer pass — NOT started):
  1. Build/point the SCRAPER CALLER: it scrapes TM, resolves games to events, and POSTs
     them to ingest-game-results with `Authorization: Bearer <ingest_secret>` (read the

     (org_id, external_game_id); unmatched games come back as reviewFlags for review.

     into THIS repo's store — the cutover replaces that with the app-as-source flow.
  2. Wire astersports.io/aau (and/or the app records surface) to get_public_team_records

     bypasses RLS, so this only guards manual UI writes).

the above and remains the single schedule writer. Map-link priority, draft-schedule
