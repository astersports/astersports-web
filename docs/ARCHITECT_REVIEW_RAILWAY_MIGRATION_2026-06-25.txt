SURFACE 2 — AUTH swap (Manus WebDev OAuth -> Google OAuth, own-JWT session)
=========================================================================================
REQUEST: ratify the identity model. Scope chosen = §3.B option (b): keep the own-JWT
  session layer, swap ONLY the OAuth provider to Google (preserves the `openId` model;
  does NOT remap to Supabase Auth user IDs). Confirm tenant-isolation is unaffected.

EVIDENCE:
  - server/_core/googleAuth.ts: buildGoogleAuthUrl (scope "openid email profile",
    state CSRF), exchangeGoogleCode -> id_token, verified against Google JWKS via jose
    (createRemoteJWKSet). No client secret/Google token reaches the browser.
  - server/_core/oauth.ts: GET /api/auth/google/login sets an oauth_state CSRF cookie
    (41-55); GET /api/oauth/callback re-checks state (56-69), derives
    openId = googleOpenId(profile.sub) (83-86), mints the app session via
    sdk.createSessionToken(openId, ...) and sets COOKIE_NAME (93-99). => identity is
    keyed on a Google-derived openId; the own-JWT session is unchanged downstream.
  - server/_core/cronAuth.ts: accepts CRON_SECRET via x-cron-secret OR Authorization:
    Bearer (plus a legacy x-webdev-schedule-uid path retained for transition); secret
    gate open only when CRON_SECRET unset (unit CI).
  - client/src/const.ts getLoginUrl -> /api/auth/google/login (Manus portal removed;
    resolved in the origin/main reconcile merge).

OPEN ITEMS FOR THE ARCHITECT:

       auth surgery riding the migration merge. (See POST-MERGE below.)
  (iii)cronAuth still accepts the legacy x-webdev-schedule-uid header — harmless on
       Railway (nothing injects it) but can be dropped with the dead-code PR.

CC LEAN [GROUNDED]: APPROVE option (b) as shipped; track (ii)/(iii) as the cleanup PR.
  Option (b) is the smaller, lower-risk choice and the live 302 + working sessions prove

SURFACE 4 — PLAN DEVIATION: in-process scheduler (vs the plan's "Railway Cron services")
=========================================================================================
REQUEST: ratify the deviation from RAILWAY_SUPABASE_MIGRATION_SCOPE.txt §2.E/P6.

EVIDENCE (server/_core/scheduler.ts):
  - In-process interval scheduler replaces Manus Heartbeat: poll-predictions every 1 min,
    reap-stuck-jobs every 5 min (27-29), self-HTTP to localhost with x-cron-secret (44).
  - Safe-by-default: no-op unless ENABLE_SCHEDULER=true AND CRON_SECRET set; refuses to
    start (logs error) if CRON_SECRET missing (56-63). Idempotency-safe: underlying jobs
    (atomic claim CAS, refund-once reaper, ON CONFLICT billing) tolerate a duplicate run.
  - heartbeat.ts and dataApi.ts removed.

RATIONALE FOR DEVIATION: on a persistent Railway instance the cold-start/scale-to-zero
  failures the plan worried about disappear, so an in-process scheduler is simpler than
  standing up separate Railway Cron services and keeps the cron handlers self-contained.

OPEN [VERIFY] (CC could not read prod env): confirm ENABLE_SCHEDULER=true AND CRON_SECRET
  are set in the Railway prod env — otherwise the reaper does NOT run and a stranded job
  is never refunded (§6 stranded-job safety). This is the one item that could silently
  bite. CC recommends verifying before onboarding regardless of the merge decision.

CC LEAN [GROUNDED]: RATIFY the in-process approach; gate onboarding on the ENABLE_SCHEDULER
  + CRON_SECRET prod-env check.
