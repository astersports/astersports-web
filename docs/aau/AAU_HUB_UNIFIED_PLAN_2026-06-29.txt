=====================================================================
1. THE JOB TO BE DONE (Frank's frame)
=====================================================================
A parent organizing tournament-weekend life — one kid, multiple kids, or one kid in
multiple programs — opens the Hub and must instantly get:
  Q1 Where do I go?            Q2 What's the weather?
  Q3 Directions / traffic?     Q4 Is there OVERLAP — do I need a babysitter?
  Q5 My WHOLE weekend, all kids, in order?

The bar: maximized AND simplified — one screen answers it without digging. The two
goals are not in tension: surfacing the cross-team answers at the top and removing
dev-console noise makes the hub both more complete and simpler.

=====================================================================
2. SCORECARD — what the LIVE home (HubHomeV2) answers today
=====================================================================
Q1 Where do I go ........... YES.  Venue + Apple/Google directions on the up-next card
                                   (NextGame.tsx). Solid.
Q2 Weather ................. PARTIAL. Honest game-TIME forecast (aster-weather
                                   forecast.ts:202-215), BUT shown on only the single
                                   soonest game; a self-contradiction bug; the better

                                   road routing (OSRM) but TRAFFICLESS, with no UI
                                   qualifier, so "leave by" can be too late on a busy
                                   Saturday (driveTime.ts:8,37-53; NextGame.tsx:131). (WS3-T)
Q4 Overlap / babysitter .... NO.  The detector is fully BUILT, TESTED, GOOD — and

Q5 Whole weekend, in order . NO.  Hero = ONE team; no combined chronological agenda;
                                   child identity modeled but dormant. (§3, WS1-B/D)

Net: excellent PER-TEAM command center; the CROSS-TEAM "organize my weekend" layer —
which is the actual JTBD — is missing from the live screen.

3. THE CORE INSIGHT — the 10/10 is mostly already BUILT, just not WIRED
=====================================================================
The hardest parts of the JTBD exist in the codebase but are routed to surfaces the live
router never mounts. Closing the gap is largely WIRING, not greenfield.

[G] ORPHAN-1 — Conflict / "babysitter" radar.
  lib/aau/conflicts.ts:86-160 (findConflicts) + components/aau/ConflictRadar.tsx are
  built and well-tested (conflicts.test.ts): detects time-overlapping games across
  DIFFERENT tracked teams, dedupes head-to-heads, drops finals/past, and splits
  same-building ("hop courts") vs different-venue ("split up?"), labeling by kid.
  Mounted ONLY in MyTeams.tsx:168. MyTeams is imported by NOTHING (grep: only its own
  def + 2 comments). Live home = HubHomeV2 (AAUBasketball.tsx:169) → zero conflict
  signal. LIMIT: simultaneity-only; no back-to-back/travel-time conflict (driveTime
  exists but conflicts.ts doesn't use it).

[G] ORPHAN-2 — Per-game weather pill across the list.
  components/aau/weather/GameWeather.tsx (honest 7-day-windowed per-game pill) and
  WeatherCard.tsx are unmounted (zero mounts; they call a separate trpc backend, not
  aster-weather). Live weather is ONLY the soonest game via NextGame.tsx, so a parent
  scanning an upcoming list sees weather on one row, not each.

[G] ORPHAN-3 — Child identity.
  trackingStore.ts:20 has a `kid` field, but track() always sets kid:null (:99) and no
  live home component reads it. pilotKids.ts:58 is a real per-child model (Frank's two
  kids, two programs) but dormant/hardcoded, unwired. So the live model groups by
  PROGRAM (myTeamsModel.ts:124) and cannot distinguish "two kids" from "one kid, two
  programs."

[G] NO cross-team agenda primitive in the live model.
  buildMyTeamsModel buckets by team→program (myTeamsModel.ts:85-133); never merges
  games into one time-ordered list. The hero is mostUrgent→ranked[0] (urgency.ts:76) =
  one team. The ET day-grouping logic needed for an agenda ALREADY EXISTS inside
  conflicts.ts (reusable).

6. OPEN DECISIONS FOR THE ARCHITECT (request + CC lean, per §2.2)
=====================================================================
D1 · Child identity (WS1-D). Activate the `kid` model so two teams group under one
     child and "two kids" vs "one kid/two programs" is distinguishable?
     CC LEAN: YES — it's the difference between a team tracker and a family weekend
     organizer, the model already exists (kid field + pilotKids.ts + ConflictRadar kid
     labels), and the JTBD is explicitly multi-kid/multi-program.
D2 · AgentConsole on authenticated parent surfaces. Remove on parent surfaces (keep on
     landing), or demote below the countdown? (H3 honesty fix happens regardless.)
     CC LEAN: REMOVE on parent surfaces — HubHomeV2 already excludes it by design (H8);
     a never-completing progress bar + decorative tabs are comprehension cost with no
     parent payoff. Keep it on discovery/landing where "watch it work" earns attention.
D3 · Weekend agenda vs hero (WS1-B). Replace the single hero with the agenda for
     multi-team parents, or stack agenda above the rich countdown hero?
     CC LEAN: STACK — keep the rich countdown for the soonest game (the proven hook),
     add the agenda above it as the "all your stops" index; one team → agenda collapses
     to just the hero.
D4 · Conflict radar scope (WS1-A). Ship simultaneity-only now, add travel-time
     (back-to-back at different venues) later?
     CC LEAN: YES, ship simultaneity now (it's built + tested); travel-time conflict is
     a fast-follow using driveTime (a real gap, but don't block the 95% case on it).
D5 · Traffic (WS3-T / Q3). Accept "typical, no live traffic" honesty qualifier as the
     answer, or invest in a traffic-aware routing provider?
     CC LEAN: qualifier NOW (cheap, honest); evaluate a traffic provider as a separate
     decision — a tournament-Saturday ETA that ignores traffic is the JTBD's weak point.
