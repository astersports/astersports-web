P0-A · NEXT-UP CARD DOESN'T WORK
  ROOT CAUSE (high confidence): the STALE TRACKING KEY. Tracking is keyed on the
  VOLATILE tournament_division_teams.id (`team_key`). A re-ingest mints fresh ids →
  the tracked team is orphaned (often to an archived duplicate) → the schedule RPC
  returns zero games → NextGame gets an empty `games` prop → the card renders blank.
  The handoff confirms this "bit Frank's phone twice" (he re-tracked an archived
  Grand Finale duplicate).
  FIX (two layers):
    1. DURABLE (architect-ruled, see PART 5): re-key ALL tracking (tracked_teams +
       localStorage) on the STABLE `resolved_key` / `external_team_key`, not the
       volatile id. Add a schedule-RPC path that resolves a stale/archived key to its
       LIVE twin so existing tracks self-heal instead of orphaning.
    2. CARD RESILIENCE: NextGame must (a) select next game = MIN(scheduled_at) where
       scheduled_at > now AND status='scheduled', timezone-correct; (b) DEGRADE
       GRACEFULLY — no upcoming game → "No upcoming games" empty state; travel/location
       unavailable → render the card WITHOUT the drive-time row, never blank; (c) be
       instrumented (log the games count it receives so the empty-prop case is visible).
  ACCEPTANCE: a freshly-tracked LIVE team shows the next-up card (countdown · venue ·
    directions); a re-ingest does NOT orphan it; a team with no upcoming game shows the
    empty state, never a blank card. Screenshot must match render 12-features #02.

PART 3 — THE PRODUCT (one page)
================================================================================
WHAT IT IS: a universal $20/mo PER-ACCOUNT tournament-tracking service. Paste a link
  or search; track ONE or MANY teams; get real-time scores, the exact predictor, and
  game-venue directions (Apple/Google/Waze) — for ANY persona (parent, coach, program
  director, fan), with an OPTIONAL verified-kid overlay.
THREE ACCESS PLANES (two independent gates):
  A PUBLIC (free, anon): browse any tournament/team — directory, standings, scores,
    bracket, predictor, schedule + directions. The funnel.
  B PAID TRACKING ($20/account, gate is_entitled): save/track teams, the multi-team
    dashboard, real-time + alerts, the predictor for your teams, navigation, self-serve
    ingest. UNIVERSAL — any team, any persona, NO child data.
  C CHILD OVERLAY (included, gate can_access_child = is_entitled AND verified-guardian,
    LIVE): assign a kid, per-child stats + film. The ONLY roster-bound, child-data part.
  INVARIANT: canAccessChildData = paidEntitlement AND verifiedGuardian, evaluated live.
  Tracking a team ≠ claiming a kid. Payment never grants guardianship.
THE ENGINE: the killer features (schedule-change alert, next-game+travel, conflict
  radar) are ONE schedule-intelligence engine — re-scrape → diff against last-known →
  derive. Build the engine once; the three killers are views on it (PART 7).
EXPERIENCE PRINCIPLES (the 10/10 bar — these govern every screen):
  T1 accuracy is the floor (no fabrication — least of all about a minor).
  T2 relevance = simplicity (DEFAULT to the user's team/kid; never make them hunt).
  T3 one nav, the user's jobs.  T4 stats → story (not a spreadsheet).
  T5 value before friction (browse/see value before pay/claim).
  T6 protect the differentiators (film + predictor get the most polish AND discipline).

  The model is sound and on-discipline: rating = opponent-adjusted avg point margin
  (one SOS pass, per-game cap ±25, by resolved_key across all public tournaments, NULL
  when no finals); win prob = logistic(0.1 · rating_gap) (~73% at a 10-pt edge); grade
  prior = 12 pts docked per grade below the division floor; the `basis` gate showing
  "—"/"projection" when there's no signal is EXACTLY right (no false precision).
  CONDITIONS:
    • CALIBRATE against outcomes once Grand Finale games land — backtest: do ~65%
      favorites win ~65%? Tune the 0.1 scale to minimize Brier/log-loss; validate the
      12-pt grade penalty against how playing-up teams actually perform. The 0.1 and 12
      are reasonable priors, NOT yet validated against results — treat them as such.
    • KEEP TWO THINGS DISTINCT ON THE GLASS: the EXACT clinch/elimination math
      (enumerated bracket outcomes — exact, provenance "enumerated exactly") and the
      ESTIMATED pre-game win % (rating-based — labeled "projection"). They must never be
      visually conflated; the exact math is a trust asset and the estimate must not
      borrow its authority. (This is also why an in-game live win-% tick stays OUT.)
