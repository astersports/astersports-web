=====================================================================
2. THE RECOMMENDATION (one product, progressive tiers)
=====================================================================
The AAU Hub is NOT a separate product from the platform — it is the FREE TIER of the
same product. A parent should never choose between two apps. The Hub is the permanent,
login-free home and the acquisition wedge; onboarding the program ADDS an org-scoped
service layer on the same home. Concretely, a 3-rung ladder:

  TIER 0 — PUBLIC (no login; the wedge; indexable):  any AAU parent, any team.
  TIER 1 — SIGNED IN (Google):                       sync + push + saved kids.
  TIER 2 — ONBOARDED PROGRAM:                         the org-native command center.

The "additional services on onboarding" Frank described = the Tier-0→Tier-2 jump.

4. DECISION REGISTER (request · evidence · options · CC lean)
=====================================================================
DR-1 · WHERE does the unified parent home live?
  Request: pick the consolidation target for the parent product.
  Evidence: shared Supabase (G1); onboarded services + parent role/RLS live in
    aster-sports; astersports-web is fundamentally the Print Studio repo with the Hub
    bolted on; the legacy site proves a no-login public reader works (G3/G4).
  Options:
    (a) Consolidate the parent home into the PLATFORM (aster-sports / astersports.app),
        keep a public login-free AAU route as the wedge.
    (b) Keep the Hub in astersports-web; light up Tier-2 via the shared Supabase +
        entitlement (no repo move).
    (c) New dedicated parent-app repo.
  CC LEAN: (a) over time — but stage it (DR-2 first). The onboarded services already
    live in the platform; do not rebuild them in the Studio repo. Non-negotiable
    constraint: the public tier stays anonymous, fast, and indexable wherever it serves.

DR-2 · IDENTITY + ENTITLEMENT linking (the core join)
  Request: how does a Tier-0/1 Google user become a Tier-2 onboarded parent, and how does
    a tracked team map to an onboarded org to trigger the unlock?
  Evidence: Hub uses Google OAuth via Supabase (anon public reads); platform uses
    `user_roles` + org-scoped RLS; AAU teams are tournament-grain (qkey), org teams are
    org-grain — the join is non-trivial.
  Options: (a) build an entitlement bridge first (thin: tracked-team→org match +
    role grant) and PROVE the unlock in place; (b) full identity merge up front.
  CC LEAN: (a) — cheapest test of the whole thesis; one bridge risked, not a migration.

DR-3 · COPPA / CHILD-DATA boundary
  Request: confirm the boundary when a public Hub and org child PII share one home.
  Evidence: the Hub deliberately keeps `entitlement.canAccessChild = false`; the platform
    holds real rosters/players. Merging makes this boundary load-bearing.
  CC LEAN: keep the gate HARD — Tier 0/1 never touch child PII; child data appears only
    in Tier 2 behind org membership + the existing COPPA gate. (Architect owns wording.)

DR-4 · DOMAIN / SEO
  Request: where does the indexable public wedge live?
  CC LEAN: keep `astersports.io/aau` public + indexable as the acquisition surface even
    if the authed app home is `astersports.app`. Don't bury the wedge behind the app.

DR-5 · CONVERSION FUNNEL ownership
  Request: replace the external LeagueApps interest-list with in-app register/pay?
  Evidence: every CTA on the live site funnels to one LeagueApps link (`brand.ts:24`);
    `Programs.tsx:3` says "checkout wires to Aster later."
  CC LEAN: YES — own the funnel; it's the free→paid bridge. Pair with the Scout lead path.

DR-6 · FAMILY FILM ROOM (the retention bet)
  Request: build the media pipeline now, or stage after cheap retention?
  Evidence: no storage/serving/capture exists; `Highlights.tsx:123` self-labels a template.
  CC LEAN: STAGE — ship Scout + push + season recap first; film room is the big bet after
    the loop is proven. (Tier-2 / premium when built.)

DR-7 · TRAFFIC vendor (= UNIFIED_PLAN D6)
  Request: TomTom vs Google Routes vs Mapbox vs stay-OSRM+qualifier.
  Evidence: OSRM is trafficless (`driveTime.ts:8`) — the one real ETA gap (Q3).
  CC LEAN: add a real provider (TomTom, or Google Routes to reuse the platform's existing
    Google geocoding billing); the WS3-T qualifier is only the stopgap. One provider also
    covers POI (food/parking) + geocoding.

DR-8 · REUSE MODEL (= UNIFIED_PLAN D7)
  Request: call aster-sports edge functions cross-app, or PORT push/ICS/share into the
    parent app?
  CC LEAN: PORT (same vendors: Resend, web-push VAPID, Open-Meteo) for deploy
    independence; revisit once DR-1 picks the consolidation target.

DR-9 · MONETIZATION (= UNIFIED_PLAN D8)
  Request: wire Aster Plus / Stripe now or stay free pre-onboarding?
  Evidence: Stripe is present in astersports-web but AAU Plus is INERT (`PlusGate.tsx`
    cosmetic; `entitlement.ts` false for all); the platform records payments by hand (no
    Stripe API).
  CC LEAN: stay free until Tier-2 proves retention; then gate premium services
    (traffic / push / film AI / multi-kid agenda) behind Plus.
