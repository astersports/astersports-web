1. THE THREE DATA TYPES IN SCOPE (this is the widening)
================================================================================
PR #1100 covered GAMES only. The directive adds PRACTICES and LOCATIONS. They have very
different sources and lifecycles — the design must treat them separately:

  GAMES      — two sub-sources (already split in #1100):
               • AAU/Zero Gravity tournaments = dynamic weekend brackets, live scores →
                 LIVE-SCRAPE from a TourneyMachine link (spike proven, §3).
               • CYO/League Play = a SET season schedule w/ occasional edits → PASTE flow
                 (existing parse-tournament-schedule), not scrape.
  PRACTICES  — a recurring, operator-set schedule. [GROUNDED] Frank: "practice schedule is
               set in the app." Lives in the platform as events (not yet confirmed which
               event shape). NOT external, NOT scraped. The hub's job is to let the
               operator set/edit them once and surface them on the public sites.
  LOCATIONS  — gyms/venues with addresses for map deep-links (Apple/Google). [GROUNDED]
               platform has a `locations` table: google_maps_url, lat/lon, address columns
               (verified via information_schema query). Tournament venues also arrive in
               the TM scrape (the "Complexes" block carries venue name + address, §3).

5. THE CORE ARCHITECTURAL DECISION (this brief's main ask)
================================================================================
The directive says "use the AAU section as the hub that feeds the app + Legacy." There is
a real seam to resolve: the AAU section lives in astersports-web (its own Supabase,
eyitivulszubtdfpsbdx); the app + Legacy read from the aster-sports platform Supabase
(vrwwpsbfbnveawqwbdmj). "Hub" can mean the UI surface OR the data source-of-truth — and
those can live in different repos. They must be reconciled or the three surfaces drift.

--------------------------------------------------------------------------------
D-HUB — Where is the SOURCE OF TRUTH, and what is the AAU section's role?
--------------------------------------------------------------------------------
REQUEST: ratify the hub topology before any redesign code.
EVIDENCE: §2.1 (AAU has its own scraper/Supabase; Legacy already reads the platform; app
  IS the platform). §2.2 (platform already has tourney_url + writers + public reads).
OPTIONS:
  (A) PLATFORM is the source of truth; the AAU section is the OPERATOR HUB UI + a public
      showcase, writing INTO the platform. The "upload a link / set a practice / pick a
      location" controls live where org auth + admin roles already are (the aster-sports
      app), and the AAU page surfaces + deep-links to them. AAU + Legacy both READ the
      platform via public RPCs. Retire the astersports-web scraper.
      REQUIRES: public read RPCs for upcoming games + practices (D5); the AAU page re-points
      from its tRPC scraper to platform reads; a clear "where the operator clicks" answer
      (app admin vs an AAU-embedded admin that calls a platform RPC).
  (B) AAU SECTION (astersports-web) is the hub of record: operator manages games/practices/
      locations there, astersports-web owns the ingestion + a DB, and it PUSHES to the app
      + Legacy.
      REQUIRES: astersports-web gains a cross-system writer into the aster-sports platform
      (or the app/Legacy read astersports-web) — a second source of truth + a sync path;
      duplicates tables the platform already has; re-introduces the drift #1100 set out to
      kill.
  (C) HYBRID: platform is the data source of truth (A), but the AAU section hosts an
      embedded admin surface that writes to the platform via authenticated RPCs — so the
      operator's "hub" experience is literally the AAU page, while the data stays single-
      sourced in the platform.
      REQUIRES: an auth bridge from astersports-web → aster-sports for the admin writes
      (service path or user-JWT), plus the D5 public reads.
CC LEAN: (A) for the DATA, optionally (C) for the UX. The platform already owns the tables,
  the writers, the public-read gates, the org auth, and the multi-tenant model; Legacy
  already consumes it cleanly. Making astersports-web a second source of truth (B) rebuilds
  what exists and re-opens drift. "AAU as the hub" is best read as: the AAU page is the
  operator's command surface + the flagship public showcase, but it reads/writes the
  PLATFORM. Whether the admin controls are embedded in the AAU page (C) or kept in the app
  admin (A) is a UX call for you/Frank — the data topology is the load-bearing decision.
