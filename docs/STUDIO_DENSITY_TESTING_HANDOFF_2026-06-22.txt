3. THE DENSITY TEST LOOP (what Frank does; what you watch)
--------------------------------------------------------------------------------
Path: Studio Editor -> upload a garment (allover print is the stress case) ->
auto-detect -> Density control (set a reduction %) -> Generate -> confirm cost
dialog -> generation runs.

Expected with async ON:
  1. POST /api/trpc/studio.generate returns FAST (~4s) with {async:true,
     status:"sam2_processing", jobId}. The client transitions to the polling UI.
  2. The processing UI ("Generating your result..." with an elapsed timer) shows
     while the client polls GET /api/trpc/studio.getJob every 2s.
  3. The poll-predictions cron picks up the job, runs the CPU op off-request,
     writes the variation, marks the job done.
  4. The client poll sees status=done and renders the before/after. Result is a
     v2 redistribute image (even spacing, no clustering).

FOUR WATCH ITEMS (the original test charter; how each manifests now):
  (1) NO 60s TIMEOUT. Async means the request returns in ~4s and the heavy run
      is off-request, so "Connection closed before the job finished" / "fetch
      failed at 60s" should NOT recur. If it does, see §6.
  (2) NO-OP BILLING. If the op removes nothing (removed:0), the job MUST refund,
      not charge. In the async worker, runDensityOnSegmentation returns null on a
      no-op -> failAndRefund("density no-op (removed 0)"). Verify a refund row
      lands (net zero).
  (3) EVEN THINNING, NO CLUSTERING. With redistribute=true the result must be an
      even blue-noise relayout, not erase-only gaps. (v1/densityThin clusters;
      that's why v2 is authorized.)
  (4) CREDITS DEDUCT + LEDGER ROW. A credit_ledger deduct lands at enqueue
      (refId job-<id>-a<n>, reason "deduct"); on success NO refund; on
      failure/no-op a matching refund (refId job-<id>-failed, reason "refund").
      SUM(ledger.delta) per tenant == creditBalance.


--------------------------------------------------------------------------------
4. §1 LEDGER — EXACT REFS (so you can verify a clean money path)
--------------------------------------------------------------------------------
- DEDUCT at enqueue: studio.generate async branch deducts per attempt with
  refId `job-<jobId>-a<n>`, reason "deduct".
- REFUND on failure/no-op/strand: refId `job-<jobId>-failed`, reason "refund",
  idempotent on (refId, reason). The async worker, the (future) webhook, and the
  reaper all use this same refId -> no double refund.
- SUCCESS: no refund; the deduct stands.
- BACKSTOP: /api/scheduled/reap-stuck-jobs refunds any job stranded in
  sam2_processing/cpu_processing past its window. This is why a stranded job is
  ALWAYS refunded, never silently charged. (Confirmed live this session: a failed
  run's -10 came back ~6 min later with the job-<id>-failed ref.)

You CANNOT query the prod MySQL directly (no Supabase MCP for this repo; the DB
is MySQL on Manus). Ledger verification is Frank's DB check OR the in-app
balance + a credit_ledger read he runs. Ask him for the row if you need it.

6. DIAGNOSTIC PLAYBOOK (the method that worked this session)
--------------------------------------------------------------------------------
When a density generation "fails" or "charges with no result," DON'T guess the
stage. Localize it first — different stages have different fixes:

KEY INSIGHT: the client processing UI (elapsed timer, "Generating your
result...") only appears AFTER the enqueue request returns sam2_processing
(async) or after the SSE "started" event. So:
  - If Frank saw the processing countdown -> the ENQUEUE SUCCEEDED. The failure
    is DOWNSTREAM (worker/cron completion or polling), NOT the enqueue.
  - If Frank saw a spinning Generate button on the controls page then an error
    toast (no countdown) -> the ENQUEUE itself hung/failed (e.g. a synchronous
    step blew the 60s cap).

ASK FRANK FOR (he has the browser + the deploy; you don't):
  1. NETWORK TAB: which endpoint, and its timing?
     - /api/trpc/studio.generate returns ~fast with a jobId -> async engaged,
       enqueue OK -> look downstream.
     - /api/trpc/studio.generate hangs ~60s then fails -> enqueue blew the cap
       (a synchronous step — locate is now bounded by #87, so suspect something
       else added to the enqueue).
     - /api/studio/generate-stream is the request -> SSE path (asyncJobs not
       effective on the client; check studio.config / the flag).
  2. HISTORY: did a result image ever land (even late)? Late = cron slow/working;
     never = completion path never ran.
  3. CRON LOGS: /api/scheduled/poll-predictions returning 200 or 403?
     - 403 with body "forbidden"  -> Gate 1 (cronSecretOk / secret header).
     - 403 with body "cron-only"   -> Gate 2 (cron session cookie / isCron).
     - 200 but job stuck           -> the worker erred; read processAsyncJob +
                                      ask for the [poll-predictions]/[async]
                                      server logs.
  4. studio_jobs row status for the jobId: stuck in sam2_processing (cron not
     completing) vs failed (worker refunded) vs done-with-no-variation.

THIS SESSION'S CHAIN (worked): processing UI showed -> enqueue OK -> job stuck
in sam2_processing -> poll-predictions logs were a wall of 403 -> Manus injects
`Authorization: Bearer <CRON_SECRET>` but our check only read `x-cron-secret`
