4. DECISION W1 — WHERE THE PACKAGE LIVES + HOW REPOS CONSUME IT  [FRANK + ARCHITECT]
────────────────────────────────────────────────────────────────────────
REQUEST: pick the hosting/publish mechanism for `@aster/*`. All options need a new
  home for the package source (the 4 product repos are separate; monorepo ruled out).
OPTIONS:
  A) GitHub Packages (npm registry, astersports scope) + a NEW repo
     `astersports/aster-weather` holding source; publish via GH Actions on tag;
     consumers add `.npmrc` (registry + token) + pinned `@aster/weather@x.y.z`.
     PROS: real semver, clean `npm i`. CONS: most tooling — registry auth token in
     every consuming repo's CI AND its deploy build env (Railway/Vercel/Manus).
  B) Git-tag dependency (NO registry) + a NEW repo `astersports/aster-weather`.
     Consumers pin `"@aster/weather": "github:astersports/aster-weather#v0.1.0"`
     (prebuilt dist committed, or a `prepare` build). PROS: minimal infra, pin-by-tag
     = the deliberate-upgrade rule (§2) for free, no registry auth. CONS: no semver
     range niceties; tag discipline manual.
  C) Monorepo — RULED OUT by the infra spec §2.
CC LEAN: **B for the pilot** — it's the smallest step that delivers cross-repo reuse
  with pin-by-tag, no registry/token sprawl across three deploy platforms; graduate
  to A if/when the @aster/* set grows. BOTH A and B need ONE thing from Frank: **authorize
  a new repo `astersports/aster-weather`** (it's outside today's 4-repo scope, so CC
  won't create it unprompted). Name/owner is Frank's call.
