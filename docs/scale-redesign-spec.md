## 1. What's built today, and its real defect

The live Scale op is a **whole-bbox repeat resampler**, not a per-motif edit:

- `server/_core/studio/ops/scaleRepeat.ts` [code @9b50771]: `scalePrintRepeat`. Factor `targetFraction = (100+percent)/100`. Resamples the entire fabric raster with pinned **lanczos3**. **Shrink (f<1) → mirror-tile** via `mirrorTileToSize`; **enlarge (f>1) → center-crop** (discards non-center content). Composites only inside the fabric raster; garment frozen; alpha preserved; byte-identical outside the mask; 1-σ Gaussian feather (`sharp.blur(1)`). Requires a raster. Deterministic. Returns `changed:false` on empty mask or `f===1`.
- `server/_core/studio/ops/tile.ts` [code @398fe9e]: `mirrorTileToSize` is deterministic **reflect/mirror** tiling (alternate tiles flipped by parity).

**The op is already a crude version of the industry-standard behavior** ("resize the whole repeat, motifs and spacing together; shrinking fits more of the repeat in view; coverage preserved" — `docs/STUDIO_OPS_SPEC.md` §0/§2.1 [code @9c023b8]). That direction is correct and is the textile standard [trade][1][2]. **Its defect is the blind fill:** on shrink it manufactures a *mirror* repeat (`mirrorTileToSize`), which is the wrong repeat style for any print that is not actually a mirror/turnover layout; on enlarge it *center-crops*, discarding pattern. A real repeat has a **lattice** (two generating vectors) and a **layout type** (full-drop / half-drop / brick / mirror); the current op ignores both. The redesign replaces the blind fill with lattice-aware re-tiling.

## 2. Target definition (unchanged in spirit, corrected in mechanism)

Scale resizes the **entire repeat by one factor `f`** — motifs **and** their spacing together — preserving coverage and layout type. Shrinking brings **more** of the repeat into view; enlarging brings **less**. This is the spec-of-record definition (§2.1/§2.2 [code @9c023b8]) and the textile standard.

**Why this is the standard, grounded:**
- A periodic print is generated by a **lattice**: a pair of linearly independent generating vectors `(v1, v2)`; the repeat tile tiles the plane under integer combinations of those vectors [2][1][15]. Scaling the print = scaling the lattice (`v1,v2 → f·v1, f·v2`) and the tile content together; everything else (layout, spacing ratios, coverage fraction) is invariant. This is exactly the lattice-manipulation operation of Liu–Lin–Hays (map the observed lattice `L` to a regular lattice `Lr` and warp the pixels) [2].
- The textile vocabulary the product already uses — full-drop, half-drop, brick — **is** this lattice with a particular offset (§4) [trade].

**Coverage preservation is the load-bearing property:** it is what keeps Scale **orthogonal** to Density (size axis vs. count axis). See §10.

## 3. Algorithm

Six stages. Stages A–C are analysis (a graduation of the dormant detector); D–F are the op.

### 3A. Repeat analysis — recover the 2-D lattice
Recover the two generating vectors `(v1, v2)` of the repeat from the fabric raster.
- **Method:** peaks of the **2-D autocorrelation surface** of the masked fabric luminance give candidate translation vectors; the two shortest linearly-independent vectors that explain the peak grid are the generating vectors [3]. Robust/automatic variants: deformed-lattice detection via MSBP + thin-plate-spline for real-world distortion [4]; PatchMatch-based lattice proposal recovering `{tx, ty, scale, rotation}` per match [5]; symmetry-group analysis to recover lattice + symmetry + representative motif jointly [1].
- **Relation to existing code — this is the precise upgrade:** `repeatGuard.ts` [code @bb10a14] and `eval/scaleMetrics.ts` [code @f632eb8] compute **per-axis 1-D** autocorrelation (project onto x and y, find a dominant period per axis). That recovers axis periods `(W, H)` but **cannot recover the offset / obliqueness**, so it cannot tell full-drop from half-drop. The literature method is the **2-D** autocorrelation surface, which recovers the full lattice including offset [3]. **Upgrade = 1-D per-axis autocorrelation → 2-D autocorrelation-surface lattice.** The dormant `repeatDetector.ts` [code @3bdefd0] already does FFT-propose + autocorrelation-validate and is the natural home for this.

### 3B. Layout classification — read the offset
Classify the layout from the recovered lattice. The second generating vector's offset component **is** the textile "drop" [trade]:

| Layout | Generating vectors (W = tile width, H = tile height) | Offset signature |
|---|---|---|
| **Full-drop / block / grid** | `v1=(W,0)`, `v2=(0,H)` | rectangular lattice; offset ≈ 0 |
| **Half-drop** | `v1=(0,H)`, `v2=(W, H/2)` | oblique; horizontal step carries **½H** vertical shift |
| **Brick / half-brick** | `v1=(W,0)`, `v2=(W/2, H)` | vertical step carries **½W** horizontal shift |
| **Mirror / turnover** | not a pure translation lattice — reflection/glide symmetry | detect via reflection symmetry, not translation alone |

- 50% is the dominant offset in practice but any fraction occurs (third-drop, etc.) [trade]; classify by the measured offset ratio, don't assume ½.
- Mirror/turnover is a **symmetry group with reflection**, not a translation-only lattice; detect it with the symmetry-group machinery [1], not by offset alone.
- **v1 supports full-drop + half-drop.** Brick + mirror are fast-follows after calibration (decision §12).

### 3C. Router — "no restrictions" (no hard reject)
Under "best-in-class, no restrictions," the analyzer is a **router**, not a reject gate:
- **True repeat** (lattice recovered, ≥ N tile repeats visible) → **lattice path** (§3D).
- **Single placed graphic / aperiodic** (no lattice) → **resize-about-center path**: scale the one graphic about its centroid; no re-tiling. (This is the only sane meaning of "scale" for a placement print.)
- **Border / one-directional repeat** → **frieze path**: a frieze is periodic along one axis only (7 frieze groups) vs. a wallpaper pattern periodic along two (17 wallpaper groups) [1]. Scale along the repeat axis; treat the cross-axis as placement.

This replaces today's hard `NON_REPEAT_SCALE` reject (`aiEngine.ts` [code @d61931b], guard from `repeatGuard.ts` [code @bb10a14]) for the single-graphic case: instead of refusing, route to center-resize.

### 3D. Lattice path — scale the unit and re-tile on the scaled lattice
1. Extract one **unit tile** at the recovered lattice (the fundamental domain).
2. Scale the tile content **and** the generating vectors by `f` (`v1,v2 → f·v1, f·v2`) — anti-aliased on shrink (§3E). This is the `L → Lr` warp of [2].
3. **Re-tile** the fabric region on the **scaled lattice**, preserving the **original phase/offset** so the repeat stays locked to where it was (no visible jump at the mask boundary). Layout type from §3B is preserved by construction (the offset scales with the lattice).
4. **Seam handling** for tiles that are not perfectly periodic (real prints rarely are): join tiles with a **minimum-error boundary cut** (image quilting [6]) or **graph-cut** seam [7]; for layouts with reflection symmetry, symmetry-guided synthesis keeps the seam consistent with the group [8]. This is the principled replacement for `mirrorTileToSize` [code @398fe9e].

### 3E. Anti-aliased resample
- **Shrink (f<1)** undersamples and will **alias/moiré** on fine prints unless prefiltered. Standard fix = prefilter before decimation; content-insensitive filters (subsampling vs. bicubic/**Lanczos**) trade aliasing against blur [11]. The repo's pinned **lanczos3** [code @9b50771] is a reasonable prefilter and an acceptable v1 choice.
- **Best-in-class upgrade path** (post-v1): content-adaptive downscaling (bilateral space+color kernels, crisper without ringing/aliasing) [11]; perceptual SSIM-optimized downscaling [12]; classic reconstruction-filter design [13]; (see also "Rapid, Detail-Preserving Image Downscaling," ACM TOG 2016 [14]). **Recommendation:** keep lanczos3 for v1, flag content-adaptive as a measured follow-up.
- **Enlarge (f>1)** upsamples — no aliasing risk, but resolution can fall below print DPI; the DPI guard (§6) governs this.

### 3F. Composite
Reuse the existing, proven compositing path — **no new compositing code**:
- 1-σ Gaussian feather at the mask edge [code @9b50771].
- Byte-identical outside the mask; garment frozen; alpha preserved [code @9b50771].
- Where survivor pixels must be preserved exactly (relevant when chained with Density), reuse the **survivor-clip / byte-identical-restore** pattern and ground-infill helpers from `densityThin.ts` [code @fed620b] (`baseClothAnchor`, `infillBaseCloth`, `markInstance`, `dilate`).

## 4. Why enlarge never overlaps under this design (the argument that fixed the direction)

A prior direction was "motifs change size but **not** position" (resize each motif in place at fixed spacing). That is **incompatible** with "enlarge must not overlap": enlarging motifs at fixed spacing must eventually collide. Avoiding overlap **requires spacing to grow with size** — which is precisely scaling the lattice, i.e., this design. Under repeat-scale, enlarge scales `v1,v2` by `f` too, so inter-motif spacing scales with motif size and **overlap cannot occur** at any `f`. This is also why the period/lattice-ratio metric (§5) is the correct estimator here (it is not, under fixed-position resizing).

## 5. Metric & acceptance

Keep the spec-of-record pass bar and add an orthogonality check:
- **Primary — lattice-vector-length ratio.** `f_achieved = ‖v1_out‖ / ‖v1_in‖` (and `v2`), compared to target `f`. This is the 2-D generalization of the period estimator that `scaleMetrics.ts` already calls primary (autocorr out/in ratio) [code @f632eb8, Architect ruling R3].
- **Global cross-check — log-polar scale.** Reddy–Chatterji: the log-polar transform of the Fourier magnitude turns scale into a shift recoverable by phase correlation [9][10]; gives a whole-image `f` estimate independent of the lattice fit.
- **Orthogonality invariant — coverage-delta ≈ 0.** Scale must not change motif coverage fraction (that's Density's axis). Measure `|coverage_out − coverage_in|`; require ≈ 0. **This is the new check** that protects the Scale/Density split (§10).
- **Unchanged pass thresholds** (match §2.4/§8 [code @9c023b8] and `scaleMetrics.ts` [code @f632eb8]): `scaleRatioError ≤ 0.15`, `paletteDeltaE ≤ 5` (reuse `fabricPalette` kmeans-5 LAB), `poseBgDeltaE ≤ 2` **excluded** from pass. Metric confidence = MIN across axes (the conservative reading the metric already uses).

## 6. Guards retained / changed

- **Empty raster → NO_OP.** Keep (`aiEngine.ts` [code @d61931b], `scaleRepeat` `changed:false` [code @9b50771]).
- **Upscale DPI guard.** Keep `checkUpscaleDpi` [code @b8040f5]: enlarge-only, `effectiveDPI = sourceDPI / f`, reject pre-deduct when `< MIN_EFFECTIVE_DPI = 150` if DPI present, warn-only if absent. (Spec §2.x lists this as *planned*; it is **built and wired** — delta D-ii, §14.)
- **Shrink min-feature advisory.** `checkScaleDownAdvisory` (min-feature `MIN_FEATURE_INCHES = 0.02`) **exists but is unwired** [code @b8040f5]. **Recommendation:** wire it on the shrink path so over-shrink that destroys fine detail is flagged (pairs with §3E aliasing).
- **Hard non-repeat reject → router.** Today `checkRepeat < 0.2 → NON_REPEAT_SCALE` pre-op [code @d61931b, @bb10a14]. Under "no restrictions" this becomes the §3C router (single graphic → center-resize) rather than a refusal.
- **Async honest-message gap (fix).** The friendly `NON_REPEAT` user message lives in the **sync** failure block, but deterministic scale runs **async** (fire-and-forget; frontend polls) and its catch stores the raw `"NON_REPEAT_SCALE"` as the job error [code @97bdf59]. With the router replacing the reject this path mostly disappears, but any remaining thrown errors on the async path still need a friendly mapping — verify against the frontend poller.

## 10. Composability — Scale ↔ Density

**Two orthogonal axes.** Scale moves the **size** axis (motif size = `f` × original; coverage preserved). Density moves the **count** axis (remove `p%` of motifs; surviving motif size preserved). Independent by construction; the spec already treats them as orthogonal, run one-at-a-time in v1 (§4 [code @9c023b8]).

**They commute (proof).**
- Repeat-scale preserves coverage and motif count-in-view-density per unit tile; it changes motif size to `f·s`.
- Density preserves motif size; it removes a fraction `p`, leaving coverage `(1−p)·C`.
- **scale → density:** size `f·s`, then coverage `(1−p)·C`.
- **density → scale:** coverage `(1−p)·C`, then size `f·s`.
- Both orders land at **motif size `f·s`, coverage `(1−p)·C`** — identical end state. The operations commute because each touches a different axis and neither's invariant is the other's variable.

**Canonical chain order: scale → density.** Rationale (not correctness — they commute — but cleanliness): Scale re-tiles a clean, well-formed repeat first; Density then removes a percentage of the **re-segmented** instances on the scaled lattice, so removal operates on the final motif geometry. Re-segment instances after Scale before running Density.

**Shared machinery makes chaining cheap:** both ops already share instance masks [code @89e4452], ground infill + survivor-clip [code @fed620b], feather + byte-identical compositing [code @9b50771], and palette ΔE [code @f632eb8]. Chaining is orchestration over shared parts, behind a flag — consistent with "Frank flips one flag at a time."
