2. DR-A — INGEST: schedule pipeline ≠ scores pipeline  [STOP-AND-FLAG, v2 §3]
────────────────────────────────────────────────────────────────────────
EVIDENCE: parse-tournament-schedule (edge fn) PARSES schedule text and RETURNS rows;
  it does NOT write. The COMMIT is client-side in useImportSchedule.js:111-142 and
  writes ONLY to `events` (insert/update), never to `game_results`. So the existing
  "one pipeline" is schedule→events. The scraper's payload includes SCORES, which
  have no home in that pipeline.
IMPLICATION: B1 ("ingest = a trigger in front of the existing parser; one write
  path") holds for SCHEDULE but NOT for SCORES. Scores need their own ingester.
  Per v2 §3 ("if the existing parser can't accept P1's payload without forking,
  STOP and flag"), CC is flagging.
OPTIONS (Architect to rule):
  A1) TWO converged paths, not one:
      (i) SCHEDULE → reuse parse-tournament-schedule → events (unchanged).
      (ii) SCORES → a NEW service-auth edge fn `ingest-game-results` (app_secrets
           Bearer, verify_jwt=false) that, per scraped game: match the EVENT
           (team_id + start_at + opponent, match-only → REVIEW FLAG if none, per
           B2), match-or-create the OPPONENT (B2), then UPSERT game_results on
           (org_id, external_game_id). Mirrors briefing-auto-draft-tick's structure
           + the app_secrets auth. NOT a fork of the schedule parser — a sibling for
           a concern the parser never had.
  A2) Schedule-only this pass: ingest only schedule via the existing parser now;
      defer scores ingest to a later spec. (Public /aau shows schedule + the
      records already in game_results, but no NEW scraped scores until A1 lands.)
CC LEAN: **A1** — it's the faithful realization of "app owns scores" (R3) and keeps
  one write path PER CONCERN (schedule events; score results), with the same auth.
  It's a sibling ingester, not a parser fork, so it respects B1's intent. A2 is the
  truer pick only if Frank wants to ship the portal on existing records first and
  add live-score ingest later.

3. DR-B — PUBLIC READ is RLS-gated, not a column GRANT (A4 wording)
────────────────────────────────────────────────────────────────────────
EVIDENCE: there is NO direct `GRANT SELECT (cols) ON game_results TO anon`. Public
  visibility is an RLS POLICY `game_results_select_public` (20260612030336) =
  `USING (published_at IS NOT NULL AND EXISTS(event))`. The "allowlist" is therefore
  (a) the RLS row filter + (b) the columns get_public_team_records SELECTs.
RESOLUTION (no fork needed): build get_public_team_records as a SECDEF RPC gated by
  org_is_public_listed(org_id) that SELECTs ONLY the display columns
  (id, event_id, our_score, opponent_score, result, point_differential,
  coach_highlight, published_at) for published rows — NEVER published_by/private_notes/
  org_id. Functionally identical to A4's intent; the doc should say "RLS policy +
  RPC column projection" instead of "reuse the anon column grants."
