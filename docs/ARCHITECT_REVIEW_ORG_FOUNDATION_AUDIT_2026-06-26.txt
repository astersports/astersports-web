2. WHAT PR #129 BUILDS (step 2 — the append-only audit trail, spec §8)
--------------------------------------------------------------------------------
SCHEMA (drizzle/schema.ts `platformAuditLog` + migration 0003_aberrant_santa_claus.sql):
  table platform_audit_log:
    id serial PK · actorUserId int NOT NULL (the super-admin) · action varchar(64)
    NOT NULL · targetTenantId int NULL · targetUserId int NULL · summary varchar(512)
    NOT NULL · metadata jsonb NULL · createdAt timestamptz default now() NOT NULL.
  indexes: (createdAt,id) for the global newest-first strip · (targetTenantId,
    createdAt) · (actorUserId,createdAt) · (action,createdAt).
  - action is varchar (NOT an enum) to mirror credit_ledger.reason — a new action
    needs no migration.
  - Insert-only BY CONSTRUCTION: the app exposes a writer + a reader and NO
    update/delete path. (DB-level REVOKE UPDATE/DELETE on the table is a deliberate
    FOLLOW-UP, see §5 — flagged, not silently skipped.)

WRITER/READER (server/auditLog.ts):
  - writeAuditLog(entry): append one row. BEST-EFFORT — it NEVER throws into its
    caller. Rationale: an audit-write failure must not roll back or block a
    legitimate super-admin action; we prefer "action succeeds + logged audit gap"
    over "action fails because logging failed." Failures go to the server logger.
    Returns true/false.
  - listAuditLog({targetTenantId?, limit}): newest-first read for the console strip.
  - AuditAction union: org_provisioned | individual_invited | impersonation_started
    | domain_lock_changed | org_suspended | ownership_transferred | credit_grant |
    credit_revoke. (The last four are defined now but only emitted as their features
    land.)

WIRING (server/routers/platform.ts):
  - Audits the THREE existing super-admin write actions as EVENTS: provisionFirm →
    org_provisioned; inviteIndividual → individual_invited; impersonate →
    impersonation_started. Each is a pure additive insert AFTER the existing
    operation; NO credit/ledger logic was changed.
  - Adds `platform.auditLog` read query (superAdminProcedure-gated) for the console
    audit strip (renderings Scene 1).

TEST (server/auditLog.test.ts): locks the load-bearing invariant — writeAuditLog
  returns false and does NOT throw when the DB is unavailable OR the insert rejects;
  returns true + inserts the expected row shape on success.

VERIFICATION: `pnpm run check` (tsc) clean; full server vitest 596 passed / 20
  skipped locally. CI on the PR: typecheck+tests + money-path integration.

3. WHY THIS IS SAFE TO LAND AHEAD OF H1 (money-path boundary)
--------------------------------------------------------------------------------
This PR changes NO credit/ledger logic. The audit rows on provision/invite record
the EVENT (who provisioned what, with what initial-credit parameter) — they do not
add, move, or gate a single credit. The dedicated credit ISSUE/REVOKE RPCs and
their audit rows (reason taxonomy admin_grant/admin_revoke/trial_grant, clamp-at-0,
override) remain HELD for the H1 credit-ledger sign-off (spec §5 prerequisite).
So #129 is auth/observability substrate, not a money-path change — but it is still
DB-schema + super-admin path, hence Architect-scoped and held for your sign-off,
not auto-merged.
