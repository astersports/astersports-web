2. THE CORE PROBLEM  [INFERRED, from §1 grounded facts]
────────────────────────────────────────────────────────────────────────
The SAME AAU program is implemented THREE times:
  (a) P1 astersports.io/aau  — "Aster AAU 11U Girls" (scraper-backed)
  (b) P3 legacy-hoopers site — "Legacy Hoopers 11U Girls" (scraper-backed + chatbot)
  (c) P2 astersports.app     — Legacy Hoopers as tenant #1 (system of record)

(a) and (b) are the same team (11U girls, 5th grade, Zero Gravity, jerseys #5/#24)
re-skinned under two brands. (c) is where that program's real roster/schedule/
registration data will live. Three codebases, three designs, three data stories
for one program. Without an IA ruling this drifts: scores diverge between scraper
copies, registration splits (LeagueApps vs app /r/:slug), and a visitor can't tell
how astersports.io, the program site, and the app relate.

Secondary problem: BRAND + DESIGN FRAGMENTATION across four dark/navy-ish themes
with four accent systems (P1 gold/orange, P3 cobalt, P2 per-org light, P4 emerald).
Aster (platform) and Legacy Hoopers (program) are DIFFERENT brands — but P1's /aau
currently wears "Aster AAU," blurring whose program this is. That ambiguity is
itself a fork (F3).

The forks in §3 are the decisions that resolve both problems.

── F1. Where does the AAU PROGRAM PRESENCE live? (the load-bearing fork) ──
REQUEST: Define the canonical home of the AAU program's public presence, and what
  each of the three AAU surfaces (P1/aau, P3 site, P2 public pages) is FOR.
EVIDENCE: P2 already ships PublicSchedulePage (/schedule/:teamId) + /r/:slug
  registration gated by organizations.is_public_listed [GROUNDED]. P1/aau and P3
  both re-implement scores/records/film off the same Tourney Machine scrape
  [GROUNDED]. Frank: P3 is the redesign target AND first pilot on P2.
OPTIONS:
  A) App-as-home. The program's living data (schedule, roster, records,
     registration) is served by P2's public surfaces in Legacy Hoopers branding.
     P3 (legacy-hoopers site) is redesigned as a thin marketing shell that links
     INTO the app's public pages (or is retired, see F6). P1/aau becomes a
     short CASE STUDY on astersports.io linking out — not a full duplicate.
     Requires: enrich P2 public surfaces (public records/film/about, not just
     schedule); set is_public_listed; a public program landing route on P2.
  B) Standalone-site-as-home. Redesign P3 as the program's full public site;
     P2 is the parents' logged-in operations app; P1/aau is a case study. The
     scraper stays the program-site data source; the app owns ops only.
     Requires: redesign P3; define scraper-vs-app data boundary (F2); P1/aau
     demoted to teaser.
  C) Status quo+. Keep all three, hand-define non-overlapping scopes + a shared
     design kit + shared data source so they don't drift. Most surface area to
     maintain; highest drift risk.
CC LEAN: **A (App-as-home).** P2 is the only one of the three that is the actual
  system of record, already has public infra, and is where Frank wants the pilot.
  Two scraper re-implementations of the same scores (P1/aau, P3) are the drift
  liability AP#63 (PATTERN A: one concept, many sources) warns about. Make the
  app the single source of program truth; let astersports.io and any program
  marketing shell be thin presentation that links in. (Hedge honesty per AP#39:
  if the program needs a richer editorial/recruiting site than the app's public
  pages can be, B is the truer answer — Frank's call on how "brochure-rich" vs
  "live-data" the public presence should be.)

── F2. Canonical DATA SOURCE for scores / records / schedule ──
REQUEST: One source of truth for game scores, records, and schedule across all
  surfaces that show them.
EVIDENCE: P1/aau + P3 scrape Tourney Machine (server/scraper.ts, 5-tournament
  registry) [GROUNDED]. P2 is the system of record for schedule/roster but does
  NOT scrape live tournament scores today [GROUNDED]. PR #106 (open) adds venue
  coords + Open-Meteo weather to P1's scraper.
OPTIONS:
  A) App owns it: move/мirror the scraper into P2 (or an edge function) so the
     app holds scores+records; all public surfaces read the app. Requires
     scraper port + a public read API/RPC on P2.
  B) Scraper service owns it: one shared scraper service feeds all surfaces; the
     app ingests from it for the program's records page.
  C) Keep per-surface scrapers (status quo) — explicitly rejected by AP#63.
CC LEAN: **A**, conditioned on F1=A. If the app is the program home, the app
  should hold the scores. Near-term: keep P1's scraper running (it's live) but
  treat it as a temporary source feeding the app, not a parallel truth. Pin the
  cutover to after the pilot's app schedule is populated.

── F3. BRAND MODEL: Aster (platform) vs program/tenant brand ──
REQUEST: A rule for WHEN a surface wears Aster's brand vs. the program's own
  brand — and resolve "Aster AAU" vs "Legacy Hoopers."
EVIDENCE: P2 already encodes the right model: organizations.brand_colors applied
  at runtime; Legacy Hoopers override = cobalt #4a8fd4 [GROUNDED]. P1/aau is
  branded "Aster AAU 11U Girls" while P3 brands the same team "Legacy Hoopers"
  [GROUNDED]. Aster mark = gold/orange constellation; Legacy = cobalt knight.
OPTIONS:
  A) Platform-vs-tenant split (mirror the app): Aster brand (gold/orange) wears
     ONLY platform chrome — astersports.io marketing, Studio, Platform Console.
     A PROGRAM always wears ITS OWN brand (Legacy Hoopers cobalt + knight),
     everywhere it appears (app public pages, program site, and the case-study
     card on astersports.io). "Aster AAU" is retired in favor of the program's
     real identity, unless Frank is deliberately launching a NEW house program
     called Aster AAU distinct from Legacy Hoopers.
  B) Aster-forward: the program is presented under Aster's brand as a flagship
     ("Aster AAU"), Legacy Hoopers name de-emphasized.
CC LEAN: **A**, pending one Frank input: is "Aster AAU 11U Girls" (a) just a
  re-skin of Legacy Hoopers' 11U girls that should revert to Legacy branding, or
  (b) a genuinely NEW Aster-owned house team? The whole AAU section's identity
  hinges on that one answer. Default lean = (a) revert to Legacy Hoopers branding;
  it matches the app's tenant model and avoids inventing a second program identity
  for one team.

── F4. MAIN-PAGE IA: agency-studio vs product-hub framing ──
REQUEST: What IS astersports.io — a design/tech studio that happens to have
  products, or a product platform that also does agency work?
EVIDENCE: Home.tsx today reads as a studio/agency site; PR #107 added a registry-
  driven Platform section that lists products (Studio, App, AAU) AND keeps agency
  services [GROUNDED]. The registry supports either framing without a rebuild.
OPTIONS:
  A) Product-platform-first: hero + primary nav lead with the products (Studio,
     the Sports App, programs); agency services become a secondary "we also build"
     band. Best if the SaaS app is the growth bet.
  B) Studio/agency-first: lead with the studio story; products are showcased work.
     Best if near-term revenue is project/agency work.
  C) Audience-split entry: a top-level chooser (two doors: "For sports orgs" →
     app/programs, "For brands" → studio/agency), each with its own narrative.
CC LEAN: **A (product-platform-first)**, with the agency services retained as a
  band (already built). The monetized, scalable assets are Studio + the app; the
  registry already makes products first-class. C is attractive if the two
  audiences barely overlap — flagging it as the truer option IF Frank sees the
  brand/sports buyers as fully distinct funnels.

── F5. CROSS-PROPERTY DESIGN LANGUAGE ──
REQUEST: One shared design language across Aster-owned surfaces, or intentionally
  distinct per property?
EVIDENCE: Four accent systems today [GROUNDED]. Drive defines the canonical Aster
  gradient E0631C→E8902A→F6CC55→FBD56B; P1 uses #f5b731→#e67e22 (close, not exact)
  [GROUNDED]. P2 has a locked --as-* token system already.
OPTIONS:
  A) One Aster design kit for platform chrome (astersports.io, Studio, Console):
     align P1's gradient to the Drive brand spec; shared type scale, tokens,
     components. PROGRAMS keep their own brand inside their themed pages (per F3).
  B) Per-property bespoke (status quo) — maximizes distinctiveness, maximizes drift.
CC LEAN: **A.** Unify Aster platform chrome to the documented brand (fix the
  gradient first — cheap, high-signal), keep tenant/program theming as the
  deliberate exception driven by brand_colors. This is the only fork with an
  immediate low-risk action (gradient alignment) I can take on request.

── F6. FATE of the legacy-hoopers repo (P3) ──
REQUEST: Redesign-in-place, fold into the app's public surface, or retire?
EVIDENCE: P3 is a full standalone site; P2 can already render a public program
  presence; Frank wants P3 redesigned AND the program piloted on P2.
OPTIONS:
  A) Fold-in: retire P3 as a separate app; the program's public presence becomes
     P2 public pages (Legacy-branded) + optional thin marketing shell. One less
     codebase to maintain. (Pairs with F1=A.)
  B) Redesign-in-place: P3 stays a separate marketing site, redesigned, reading
     program data from the app/scraper. (Pairs with F1=B.)
CC LEAN: **A if F1=A; B if F1=B.** These two forks move together — F1's ruling
  decides F6. Lean A overall (fewer codebases, app is the home).

5. CROSS-CUTTING STANDARDS (adopt platform-wide)  [GROUNDED in the St Pat review]
────────────────────────────────────────────────────────────────────────
The St. Patrick Architect review already enumerated these; they apply to P1/P2/P3:
  • Mobile persistent bottom-nav for primary destinations (P0 in that review).
  • SEO: per-page meta/OG/Twitter cards + JSON-LD structured data (Event,
    SportsTeam/Organization), dynamic sitemap/robots. astersports.io has an
    ogimage.png in Drive ready to wire.
  • A11y: 44px tap targets, WCAG AA contrast on hero-over-image, reduced-motion,
    aria labels — applies to P1 hero + AAU pill nav.
  • Perf: route-based lazy-loading, image optimization (WebP/AVIF + srcset),
    stale-while-revalidate caching for scraper/ICS feeds.
These are routine (non-architectural) and can ship incrementally once the IA
forks land; listing them so the Architect can fold them into the design pass.
